# webpack 是如何运作的

webpack 运作的大体思路是：

  • 先找到入口文件，并解析入口文件；

  • 找到入口文件的依赖(其它 import 文件)，并将入口文件的代码进行转换；

  • 使用递归的方式一层一层地将依赖与依赖中的代码进行解析，直到最后没有依赖为止；

  • 最终通过将所有代码解析为浏览器能够运行的代码。

# 模块解析

首先，做第一步和第二步：

const fs = require('fs');
const path = require('path');
const parser = require('@babel/parser');
const babel = require('@babel/core');
const traverse = require('@babel/traverse').default;

const moduleAnalyser = filename => {
  const sourceCode = fs.readFileSync(filename, 'utf-8');
  // babel 解析源代码（只有 sourceType 为 module 时才能解析 import export）
  const AST = parser.parse(sourceCode, {
    sourceType: 'module'
  });
  const dependencies = {};
  // 在 AST 的 node 树上做转换
  traverse(AST, {
    ImportDeclaration({node}) {
      const current_dirname = path.dirname(filename);
      const dependency_path = path.join(current_dirname, node.source.value);
      dependencies[node.source.value] = dependency_path;
    }
  });

  // babel core 将 ast 抽象语法树转换为浏览器能够读得懂的代码 (es6 -> es5)
  const { code } = babel.transformFromAst(AST, null, {
    presets: ['@babel/preset-env']
  });

  return {
    filename,
    dependencies,
    code,
  };
};

解析的步骤大概是：

  • 将入口文件路径作为参数传入 moduleAnalyser 函数中，通过 node 的 fs 模块将入口文件中的代码转化为字符串；
  • 使用 @babel/parser 将文件中的字符串转化为 AST 语法树；
  • 使用 @babel/traverse 进一步解析 AST 语法树，将其中依赖的路径进行解析拼接；
  • 使用 @babel/core 与 @babel/preset-env 两大 babel 核心功能将整个 AST 语法树转换为浏览器能够读懂的代码（字符串），最后返回一个对象，包括了文件路径、文件中的依赖与文件可读代码。

# 深层递归

然后我们做第三步，将文件中的递归一层一层进行解析：

// 将入口文件的所有依赖全部解析，解析过程同上
const deepModuleAnalyser = entry => {
  const entryModule = moduleAnalyser(entry);
  const deepModuleArr = [entryModule];
  // 递归解析其中的每个依赖
  for (let i = 0; i < deepModuleArr.length; i++) {
    const _module = deepModuleArr[i];
    if (_module.dependencies) {
      Object.keys(_module.dependencies).forEach(index => {
        const dependency = _module.dependencies[index];
        deepModuleArr.push(moduleAnalyser(dependency));
      });
    }
  }
  
  // 转换为键值对，处理更加友好
  const graph = transform2Graph();
  return graph;
};

const transform2Graph = () => {
  const graph = {};
  deepModuleArr.forEach(deepModule => {
    const {
      filename,
      dependencies,
      code
    } = deepModule;
    graph[filename] = {
      dependencies,
      code
    }
  });
  return graph;
};

console.log(deepModuleAnalyser('./src/index.js'));

大致的步骤为：

  • 声明一个最终需要的依赖数组 deepModuleArr，通过解析入口文件作为其第一个元素，进行初始化；
  • 循环读取 deepModuleArr，将其中的每个元素进行解析，找到每个元素的 dependencies，每解析一个元素，将结果 push 到 deepModuleArr，后续每新 push 的元素都是后续的依赖，从而实现递归。

# 生成代码

最后需要将以上信息转换为浏览器运行的代码。

将以上对象中的代码转化为字符串，并使用 eval 去执行该段代码，需要注意的是，依赖中的路径通常是类似 ./a.js 等，需要将其转换为绝对路径。因此使用了以下闭包函数来封装了一套 require 来获取绝对路径下的依赖：

const generateCode = (entry) => {
  const graph = JSON.stringify(makeDependenciesGraph(entry));
  return `
    (function(graph){
      function require(module) {
        function localRequire(relativePath) {
          return require(graph[module].dependencies[relativePath]);
        }
        var exports = {};
        (function(require, exports, code){
          eval(code)
        })(localRequire, exports, graph[module].code);
        return exports;
};
      require('${entry}')
    })(${graph});
`; }

最终浏览器只要跑以上代码就可以运行入口文件 index.js 中的代码。

# 手写loader

# 基本操作

Loader 就是一个声明式函数，不能用箭头函数。

将源代码作为参数，做进一步的修饰处理，再将处理后的源码返回即可。

举个例子，我们的需求是将所有 js 文件中的 world 转换为对应 webpack.config.js 中的字符串。

新建 loaders 文件夹：

project/
--| loaders/
-----| replace.js

写一下该 loader 的处理方法：

module.exports = function(source) {
  const { word } = this.query;
  return source.replace('world', word);
}

最后配置一下 webpack.config.js:

module.exports = {
    rules: [
        {
            test: /\.js$/,
            use: {
              loader: path.resolve(__dirname, '../loaders/replaceLoader.js'),
              options: {
                word: 'xhz'
              }
            }
        },
    ]
}

这里 options 其实就是 loader 的参数，最后在 loader 文件中以 this.query 中取到该参数。

loader 必须返回经过处理后的源代码，可以是 string 或 buffer。

可以看到打包后的文件中所有的 world 都被替换为 xhz

官方文档 接口文档

# this.callback

我们同样可以使用 this.callback 返回代码。

this.callback(
  err: Error | null,
  content: string | Buffer,
  sourceMap?: SourceMap,
  meta?: any
)

所以可以改写一下之前的 loader:

module.exports = function(source) {
  const { word } = this.query;
  const code = source.replace('world', word);

  this.callback(null, code);
}

# 异步编写

假设在 loader 中有一些需要异步处理的代码，比如在我们的 loader 中需要3秒后将处理后的代码进行返回，我们该怎么做？

这里使用了 this.async() 进行处理。

module.exports = function(source) {
  const cb = this.async();
  setTimeout(() => {
    const { word } = this.query;
    const code = source.replace('world', word);
    // 使用 this.async 将代码返回
    cb(null, code);
  }, 3000);
}

# resolveLoader

resolveLoader 用于直接解析 loader 的路径。

module.exports = {
    resolveLoader: {
        modules: ['node_modules', './loaders']
    }
}

resolveLoader 会先去 node_modules 中寻找对应名称的 loader，若没有则回去 ./loaders 中寻找。

那么我们的 loader 就可以不用解析路径了：

module.exports = {
    rules: [
        {
            test: /\.js$/,
            use: {
              loader: "replaceLoader",
              options: {
                word: 'xhz'
              }
            }
        },
    ]
}

# Plugin

一个插件 Plugin 实际上就是一个类，它允许你使用 webpack 的底层特性来做相应的钩子(hooks)。

# 创建第一个插件

我们的需求是创建一个在 webpack 打包后能够在我们 dist 文件夹下创建一个 copyright.txt 文件的插件。

一个插件需要的必要因素：

  • apply 方法
  • 钩子函数
  • 功能完成后调用 webpack 提供的回调

首先编写一下我们的插件:

// copyright-webpack-plugin.js

class CopyrightWebpackPlugin {
  constructor(options) {
    console.log('options: ', options);
  }

  // compiler 是 webpack 实例
  apply(compiler) {
    // emit，在创建文件之前执行的钩子
    compiler.hooks.emit.tapAsync(
      "CopyrightWebpackPlugin",
      (compilation, callback) => {
        compilation.assets["copyright.txt"] = {
          source() {
            return 'hello copyright';
          },
          size() {
            return 1024;
          }
        }
        callback();
      }
    );
  }
}
                    
module.exports = CopyrightWebpackPlugin;

我们在 webpack 执行打包后，执行了 emit 钩子，钩子用于控制我们的功能函数在 webpack 编译的什么时期执行，emit 的作用就是生成资源到 output 目录之前。

并且我们调用了 compilation —— 资源实例，直接新增一个 copyright.txt 的文件。其中 tapAsync 是异步处理。

# AST 与 Babel 原理

（以下内容摘抄自某文章）。

# Babel 工作原理

Babel 的编译过程和大多数其他语言的编译器相似，可以分为三个阶段：

  • 解析（Parsing）：将代码字符串解析成抽象语法树。
  • 转换（Transformation）：对抽象语法树进行转换操作。
  • 生成（Code Generation）: 根据变换后的抽象语法树再生成代码字符串。

为了理解 Babel，我们从最简单一句 console 命令下手。

# 解析（Parsing）

拿到源代码会把代码抽象出来，变成 AST （抽象语法树），学过编译原理的同学应该都听过这个词，全称是 Abstract Syntax Tree。

抽象语法树是源代码的抽象语法结构的树状表示，树上的每个节点都表示源代码中的一种结构，只所以说是抽象的，是因为抽象语法树并不会表示出真实语法出现的每一个细节，比如说，嵌套括号被隐含在树的结构中，并没有以节点的形式呈现，它们主要用于源代码的简单转换。

AST 是怎么来的？

整个解析过程分为两个步骤：

  • 分词：将整个代码字符串分割成语法单元数组
  • 语法分析：建立分析语法单元之间的关系

# 分词

语法单元通俗点说就是代码中的最小单元，不能再被分割，就像原子是化学变化中的最小粒子一样。

Javascript 代码中的语法单元主要包括以下这么几种：

关键字：const、 let、 var 等

  • 标识符：可能是一个变量，也可能是 if、else 这些关键字，又或者是 true、false 这些常量
  • 运算符
  • 数字
  • 空格
  • 注释：对于计算机来说，知道是这段代码是注释就行了，不关心其具体内容 其实分词说白了就是简单粗暴地对字符串一个个遍历。为了模拟分词的过程，写了一个简单的 Demo，仅仅适用于和上面一样的简单代码。Babel 的实现比这要复杂得多，但是思路大体上是相同的。对于一些好奇心比较强的同学，可以看下具体是怎么实现的，链接在文章底部。

# 语法分析

语义分析则是将得到的词汇进行一个立体的组合，确定词语之间的关系。考虑到编程语言的各种从属关系的复杂性，语义分析的过程又是在遍历得到的语法单元组，相对而言就会变得更复杂。

简单来说语法分析是对语句和表达式识别，这是个递归过程，在解析中，Babel 会在解析每个语句和表达式的过程中设置一个暂存器，用来暂存当前读取到的语法单元，如果解析失败，就会返回之前的暂存点，再按照另一种方式进行解析，如果解析成功，则将暂存点销毁，不断重复以上操作，直到最后生成对应的语法树。

转换（Transformation）

Plugins

插件应用于 babel 的转译过程，尤其是第二个阶段 Transformation，如果这个阶段不使用任何插件，那么 babel 会原样输出代码。

Presets

Babel 官方帮我们做了一些预设的插件集，称之为 Preset，这样我们只需要使用对应的 Preset 就可以了。每年每个 Preset 只编译当年批准的内容。而 babel-preset-env 相当于 ES2015 ，ES2016 ，ES2017 及最新版本。
