      # SSR vs CSR

      # CSR

      CSR(client side render): 前端渲染。

      当用户在浏览器中输入网址 url 时，服务器会去寻找对应的 html 文件(携带 js 文件)发送到前端，前端接收到 html 文件后进行初步渲染生成最初的 DOM tree。当自上朝下地渲染遇到 JS 文件时发生阻塞，先去执行 JS 文件中的代码，js 可能会对 DOM 进行操作，DOM 可能会发生 reflow 等操作，或者在 js 中发起了其它请求，被请求得到的数据返回前端后与最初的 html 进行拼接，等待 js 执行完毕后，生成最终的 html。正是因为需要等待 js 代码全部执行完毕等情况，用户在点开页面时会出现一段时间地"白屏"。 最终的 html 不一定拥有最终完整的 html 结构，可能只是部分的结构(剩余的结构由 JS 进行 dom 操作或发送请求得来)，因此不完整的 html 结构不适合做 SEO 搜索。

      特点：首屏加载速度慢，首屏加载后的其它加载速度快，适合用户交互。

      # SSR

      SSR(server side render): 服务端渲染。

      当用户在浏览器中输入网址 url 时，会在服务器中使用模板语言或 node 等方式将数据与 html 字符串进行拼接，然后将一整个拼接好的 html 返回给客户端，客户端直接渲染完整的 html。因此返回给前端的 html 是完整的，适合爬虫等工具去搜索。

      Nuxt.js 中，用户输入的 url 即是 Nuxt.js 中约定的路由，在服务端中跑了 vue.js, 解析了 template 中的内容，通过 vue 渲染出首屏内容，并返回给客户端一个 spa 脚本，而这些脚本到了前端后都是延迟加载的，正如图中所示，服务端传的首屏幕仅是 viewable, 仍然还需要在客户端下载 js 等文件并挂载 js 事件等。

      特点：首屏加载速度快，适合SEO。

      简而言之，就是数据拼接HTML字符串这件事放在服务端还是客户端造成了两者区别。

      # 神奇的 Nuxt 路由

      众所周知，在nuxt中路由跳转时，可以使用<nuxt-link></nuxt-link>标签。

      # 路由预加载

      当用户访问某个页面时，Nuxt.js 发现了页面中包含 nuxt-link，会进行预加载，即先将 nuxt-link 标签中的页面先下载下来，优点在于，在进入 nuxt-link 的页面时就不需要再加载了，加载速度更快。

      # no-prefetch

      当一些页面模块内部过大，预加载会导致当前页面加载速度慢，可使用 no-prefetch 属性，禁止预加载：

      <nuxt-link no-prefetch></nuxt-link>

      # 传参

      例如，想跳转到路由为 /main/:id 的路径时:

      <nuxt-link :to="{ name: 'main-id, params: { id: 2 } }"></nuxt-link>

      # 视图 & 布局

      在 Nuxt 中，最基础的布局是在 Layout 中。

      <template>
  <div>
    <nuxt />
  </div>
</template>

      <nuxt /> 标签中的内容就是 pages 文件中的页面内容。
